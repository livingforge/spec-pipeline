# 脅威モデル — contextdb

想定する信頼境界と、各層の防御、そして**それを検証するテスト**を対応づける文書。
関連: [GOVERNANCE.md](GOVERNANCE.md) / [dependencies.md](dependencies.md)。

## スコープと前提

- contextdb は**ローカルの仕様データ（YAML）を検証・生成・可視化するツール**であり、
  ネットワークサービスとして公開されない。ネットワークアクセス・外部モデル取得は無い。
- 想定利用者は、自組織の仕様データを管理するエンジニア／エージェント。
- したがって主眼は「**壊れた/信頼できないデータを黙って通さないこと**」
  「**生成物に仕様データ由来のテキストを安全に埋め込むこと**」
  「**正本を安全に変更すること（誤更新の抑止）**」であり、ネットワーク攻撃者対策ではない。

## 信頼境界（trust boundaries）

| # | 境界 | 内→外 / 外→内 | 主なリスク |
|---|---|---|---|
| B1 | 仕様データ (YAML) → エンジン | 外→内（利用者/エージェントが書く YAML） | 任意オブジェクト復元・破損 YAML による中断・宣言違反の看過 |
| B2 | テンプレート → 生成 HTML | 内→外（人が見る/共有しうる成果物） | 仕様テキストの HTML への未エスケープ混入・外部依存 |
| B3 | mutate → 正本 (items/ + relations/) | 内→内（正本の書き換え） | 誤更新・不整合な状態の混入・未レビュー変更の既成事実化 |

## 脅威 → 防御層 → 検証テスト

| ID | 脅威 | 信頼境界 | 防御層 | 検証テスト |
|---|---|---|---|---|
| T1 | 細工 YAML の `!!python/...` タグ等で任意オブジェクトを復元させコード実行に至る／破損 YAML で処理全体が落ちる | B1 | **D1**: 読み込みは `yaml.safe_load` のみ（`load` 不使用）。壊れた YAML は例外中断せず `error` として報告し、他の検証を継続 | `contextdb/tests`（不正 YAML の error 報告・safe_load 経路） |
| T2 | 必須属性欠落・カーディナリティ違反・重複 ID・孤児など宣言済み制約への違反を看過し、不整合な正本が下流（生成・集計）へ伝播 | B1 | **D2**: `engine` がメタモデル宣言に対し必須属性・一意性・カーディナリティ・孤児を機械検証し、error があれば exit 1 | `contextdb/tests`（各制約違反の検出） |
| T3 | 仕様データ中のテキスト（`<script>` 等）が生成 HTML にそのまま出力され、閲覧時に解釈される | B2 | **D3**: 生成 Jinja2 環境は `.html` 出力に対し autoescape を有効化。生成 HTML は CDN・外部依存なしの自己完結（`dr-05`） | `contextdb/tests`（HTML 生成・エスケープ） |
| T4 | 正本の直接編集・誤更新で不整合が混入し、未レビュー変更が既成事実化する | B3 | **D4**: 変更は `mutate` 経由に限定。新規・変更は必ず `status: review`（`approve` は人のレビュー後のみ）。適用後に再検証し**新たな error を生む plan は自動巻き戻し**（トランザクショナル） | `contextdb/tests`（mutate の review 強制・巻き戻し・approve 経路） |
| T5 | 実装だけ進んで正本が陳腐化し、CI をすり抜ける（ドリフト） | B3 | **D5**: `sync-check --strict` が error 級の乖離（unregistered / vanished / dead-path / dead-doc）で exit 1。CI / pre-commit で同期漏れを機械的に止められる | `contextdb/tests`（sync-check の検出種別・strict の終了コード） |
